## Context

当前系统不同智能体各自运行在专属的 Executor（例如数据查询智能体运行在 `DataQueryExecutor`，知识库智能体运行在 `KnowledgeExecutor`）上。目前缺乏一个通用的桥接机制，使得这些智能体能以系统工具的形式被嵌套调用，限制了主通用助手在后端进行复杂任务委派的能力。

## Goals / Non-Goals

**Goals:**
1. 提供 `sub_agent_call(agent_name, query)` 系统隐式工具，支持在主助手运行期间静默委派其他子智能体执行。
2. 实现子智能体输入上下文的完全隔离，调用时仅传入专属 System Prompt 和本次 Query，避免主会话的历史上下文污染子代理，并节约 Token。
3. 实现流式进度日志的实时穿透，将子代理在运行中产出的 `type: "log"` 消息包重写后穿透合并至主 SSE 事件流中，优化用户前端等待体验。
4. 对子代理的最终输出进行去噪清洗，剥离 `<sql_plan>` 等图形化卡片标签后再交付给主智能体。

**Non-Goals:**
1. 不支持多级代理嵌套（目前限制调用深度最大为 1，子代理中禁止继续触发 `sub_agent_call`），防止 Token 暴涨或死循环。
2. 不重构前端对话页面，子代理的日志步骤完全复用现有的前端 SSE 日志卡片折叠渲染。
3. 此版本暂不支持子智能体在执行期间向用户发起交互式挂起（如子代理执行时需要二次确认的操作，此处视为不支持或直接返回失败）。

## Decisions

### 1. 采用隐式系统工具（Implicit Tool）机制实现
*   **选择原因**：相比于直接重构 AgentScope 运行时，以 Tool 形式包装其他智能体的 Executor 能够完全融入 LLM 自身的 ReAct 推理逻辑。主助手仅需识别场景并触发 `sub_agent_call` 工具即可。此方案系统侵入性小，扩展性好。
*   **备选方案**：重构 AgentScope 运行时以支持多智能体 Pipeline 或工作流节点。该方案开发成本极高，且破坏了目前基于 Capability/Intent 进行流式调度的统一逻辑。

### 2. 构建独立且去噪的上下文隔离机制
*   **选择原因**：在拉起子智能体时，不将主智能体的全量历史 `messages` 传递给子智能体，而是为其重新构造一个精简的 `history` 列表（仅含其 System Prompt 以及主助手传来的具体 Query）。这能最大程度防止上下文交叉污染并控制 Token 开销。

### 3. 日志流的穿透与合并
*   **选择原因**：在工具内部消费子 Executor 的流式输出时，拦截其 `type: "log"` 包，将其 title 修改为 `[{子代理展示名}] {原标题}` 并重新向主 Runner 的事件流（Event Stream）中投递。这样前端能立即收到进度，防止用户在长任务中产生假死卡顿感。

## Risks / Trade-offs

*   **[Risk] 死循环与 Token 滥用**
    *   **Mitigation**: 在 `AgentContext` 中引入 `delegation_depth` 属性。在主助手调用子代理时，子代理对应的 `AgentContext.delegation_depth` 会设为 1；如果子代理试图继续触发 `sub_agent_call`，工具检测到 `delegation_depth >= 1` 将直接抛出异常拒绝调用。
*   **[Risk] 子代理执行时间过长导致接口超时**
    *   **Mitigation**: 在工具内部调用子代理 Executor 时加入 `asyncio.wait_for` 并设置超时时间（如 60 秒）。超时后捕获异常，并由工具返回友好错误信息给主助手，主助手可选择重新调用或将结果归纳为“执行超时，建议手动切换到专有助手”。
*   **[Risk] 结果体积庞大导致主助手上下文溢出**
    *   **Mitigation**: 对子代理最终返回的文本结果进行字数截断（如最大 2000 字符），若子代理返回了大型 JSON 或表格，对其进行简易去噪过滤（如移除表格结构，转为摘要）。
